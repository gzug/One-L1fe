## Doc Consolidation Proposal (Task 2)

### Current problem

Three root-level files (README, MEMORY, CHECKPOINT) cover overlapping concerns. CHECKPOINT contains durable rules. MEMORY contains some execution state. README is stale. The `docs/architecture/` directory has 15+ files, some overlapping.

### Proposed 3-file root structure

| File | Purpose | Update cadence |
|---|---|---|
| `README.md` | Public entry point. Repo orientation, structure, local setup commands. No execution state. | When repo structure changes |
| `ARCHITECTURE.md` | Stable system shape. Replaces `docs/architecture/overview.md` as the first-class architectural reference. Covers: stack, data layers, key domain contracts, compliance boundary. Links to detailed docs in `docs/architecture/`. | When architecture changes |
| `CHECKPOINT.md` | Live execution state only. Current seam, branch, PRs, blockers, next steps. No durable rules. | Every meaningful session |

`MEMORY.md` remains as the durable-decisions store, but is explicitly demoted from "startup file" to "reference when something fundamental is in question." `CHECKPOINT.md` is the sole startup file.

### What moves where

- **From CHECKPOINT "Keep" block → MEMORY.md** (already there; remove from CHECKPOINT)
- **From README "Current status" → CHECKPOINT.md** (execution state belongs there)
- **`docs/architecture/overview.md` promoted inline → ARCHITECTURE.md** (new root file, or rename overview.md and symlink)
- **`memory/YYYY-MM-DD.md` files older than 30 days → `docs/archive/memory/`**

### Immediate actions to get there

1. Delete CHECKPOINT "Keep" block (duplication with MEMORY)
2. Update README current-status to be timeless orientation only
3. Create `ARCHITECTURE.md` (or rename `docs/architecture/overview.md` and add a root symlink/redirect in README)
4. Mark `last_verified` consistently across all three files

## Screen Naming Convention (Task 6)

### Current names

| File | Tab label | What it actually does |
|---|---|---|
| `MinimumSliceScreen.tsx` | "Blood Panel" | Lab panel submission form with field-state controls |
| `WearableSyncScreen.tsx` | "Wearable Sync" | Provision + sync + compute wearables workflow |
| `LoginScreen.tsx` | (auth gate) | Auth only |

### Problem with `MinimumSliceScreen`

"Minimum slice" is an internal architecture term (the smallest complete path from input to stored interpretation). It describes the development seam, not the user-facing feature. Once a wearable screen, a trends screen, and a check-in screen exist, "minimum slice" becomes confusing shorthand that leaks internal scaffolding into the component name.

The tab label "Blood Panel" already implicitly corrects this for the user.

### Proposed naming convention

**Pattern:** `[Domain][Function]Screen.tsx`

| Domain | Covers |
|---|---|
| `LabPanel` | Blood/lab panel entry and review |
| `WearableSync` | Wearable provisioning and sync workflow |
| `WeeklyCheckin` | Self-report / context entry |
| `Trends` | Visualization and history |
| `Profile` | Settings, identity, preferences |

**Concrete renames:**

| Current | Proposed | Tab label |
|---|---|---|
| `MinimumSliceScreen.tsx` | `LabPanelScreen.tsx` | "Blood Panel" (unchanged) |
| `WearableSyncScreen.tsx` | `WearableSyncScreen.tsx` | "Wearable Sync" (unchanged) |

All controller, model, and config files follow the same prefix:
- `minimumSliceScreenController.ts` → `labPanelScreenController.ts`
- `minimumSliceScreenModel.ts` → `labPanelScreenModel.ts`
- `minimumSliceHostedConfig.ts` → `labPanelHostedConfig.ts`

**Timing:** Rename in one focused PR once the LabPanel feature is stable. Do not rename mid-seam.

## Monetisation Check (Task 5)

### Context

Monetisation target: 31.12.2026. Current state: authenticated mobile app with lab submission, wearable provisioning/sync scaffold, and hosted edge functions. No user-visible interpretation output yet.

### The gap: Tech-Demo vs Sellable Product

The product currently proves the data can flow in. It does not yet prove the data creates value for the user.

Three features cross that line:

---

**Feature 1: Interpretation output rendered back to the user**

The `save-minimum-slice-interpretation` function stores interpretation results but the app only shows raw metadata (coverageState, interpretationRunId, recommendationCount). The user never sees a readable recommendation or interpretation verdict.

This is the highest-leverage single feature. Without it, every user who enters their labs sees a JSON-adjacent debug panel — not a product.

Priority: P0. Required to call this a product.

---

**Feature 2: Wearable trend view with at least one readable signal**

The wearable sync backend is live. Daily summaries are computed. Nothing is shown to the user.

A single chart of resting heart rate or sleep duration over the last 7 days — backed by real Health Connect data — is the difference between "we store your data" and "your data does something for you." It also provides the first retention hook.

Priority: P1. Required for retention past first session.

---

**Feature 3: Weekly check-in form with personalised reflection**

The `weekly_checkins` table is in the schema draft. It is not implemented. The self-report layer (exercise, sleep, stress, energy) is what separates One L1fe's interpretation quality from a generic lab tracker: it provides the context that makes biomarker interpretation bounded and honest.

Without it, recommendations cannot distinguish "your ApoB is high and you sleep 5h/night under stress" from "your ApoB is high and everything else looks fine." The difference matters for monetisation because it is the core value proposition of the product.

Priority: P1. Required for meaningful interpretation quality and differentiated positioning.

---

### Summary table

| Feature | Gap | Monetisation relevance |
|---|---|---|
| Interpretation output to user | Exists in backend, not rendered | Blocks product-feel entirely |
| Wearable trend view | Backend live, UI absent | First retention hook |
| Weekly check-in form | Schema exists, not built | Core differentiator vs lab-only tracker |
